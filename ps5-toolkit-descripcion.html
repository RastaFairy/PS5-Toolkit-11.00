Documentación técnica

PS5 Toolkit
11.xx

Framework modular de ejecución de código arbitrario e inyección de payloads para PlayStation 5 firmware 11.00. Punto de entrada vía WebKit, persistencia en SceRedisServer.

FW 11.00 FreeBSD · AMD64 · Zen 2 C · JavaScript · Python ELF · BIN · SELF
§01 / descripción general

¿Qué es este proyecto?

Un conjunto de herramientas encadenadas que, partiendo de un bug en el motor JavaScript WebKit de la PS5, obtienen primitivas de lectura/escritura arbitraria en memoria, escalan al kernel, instalan un cargador de payloads persistente, y permiten ejecutar cualquier archivo .elf, .bin o .self en el hardware.

🕷️

Exploit de WebKit

Un bug de confusión de tipos en JavaScriptCore (variante de CVE-2021-30889 activa en FW 11.x) da primitivas de lectura y escritura arbitraria en el proceso del browser.

⛓️

ROP Chain vía Web Worker

PS5 tiene CFI forward-edge pero no shadow stack. Sobrescribimos la dirección de retorno de un Web Worker para pivotar el stack y ejecutar syscalls vía cadenas ROP.

🔑

Escalada de kernel

Desde ROP, explotamos una race condition en umtx para obtener un UAF en el kernel heap. Con eso logramos kernel R/W, root, y escape del contenedor (Jail) de Orbis OS.

📡

Loader persistente

Inyectamos el ELF loader en el proceso SceRedisServer vía ptrace. Sobrevive al rest mode y al cierre del browser. Escucha en el puerto 9021 indefinidamente.

§02 / contexto de seguridad

Protecciones del sistema y cómo se evaden

Protección Scope Descripción Estado en este exploit
SMEP Kernel Impide ejecutar código de userland desde el kernel Bypass — no ejecutamos desde userland en ring-0
SMAP Kernel Impide leer/escribir memoria de usuario desde el kernel Bypass — usamos kernel R/W primitives propias
XOM (R^X) User + Kernel Páginas ejecutables no son legibles (no podemos leer ROP gadgets) Bypass — gadgets vía sección de datos de libkernel (legible)
Clang-CFI (forward-edge) User + Kernel Valida tipos en llamadas virtuales e indirectas Bypass — atacamos el return address (backward-edge, no protegido)

§03 / cadena de explotación

Flujo completo del sistema

01

Servidor HTTP en el PC

El usuario ejecuta host/server.py. Esto sirve la carpeta exploit/ con los headers HTTP necesarios para SharedArrayBuffer (COOP/COEP) y CORS permisivo. También expone el endpoint /probe para detectar cuándo el loader está listo.

Usuario Sistema: levanta HTTP en :8000
02

Apertura de la página en la PS5

El usuario abre el browser de la PS5 y navega a la IP del PC. La PS5 carga index.html con todos los módulos JS del exploit. La interfaz muestra una barra de progreso de 5 fases.

Usuario: abre URL en browser PS5 PS5: carga los scripts del exploit
03

Fase 1 — Bug WebKit → R/W userland

primitives.js dispara el bug de confusión de tipos en JavaScriptCore. Esto retorna las funciones leakobj() y fakeobj(), con las que la clase Primitives construye read8() y write8() sobre el objeto victim ArrayBuffer. Toda la memoria del proceso WebKit es ahora accesible.

Sistema: heap spray + type confusion PS5: primitivas de memoria en WebKit
04

Fase 2 — Leak de libkernel + ROP via Worker

rop.js lee la GOT de WebKit para filtrar la dirección base de libkernel.sprx. Luego crea un Web Worker, lo "calienta" con un mensaje, y usa findWorkerStack() para recorrer la lista de pthreads del proceso y localizar su stack (tamaño = 0x80000). Finalmente sobreescribe el return address en stack + 0x7FB88 con el gadget pop rsp; ret para pivotar al ROP chain.

Sistema: ROPChain builder + stack pivot PS5: Worker toma el control del flujo
05

Fase 3+4 — Escalada de kernel + Jailbreak

kernel.js ejecuta cuatro sub-fases vía ROP: (a) kbase leak — lee un puntero del kernel desde la GOT de libkernel; (b) kernel R/W — UAF en umtx → pipe trick para kread8/kwrite8; (c) container escape — busca el proc en allproc, parchea cr_prison → prison0; (d) root — parchea ucred uid/ruid/svuid → 0 y kern.securelevel → -1. La PS5 está ahora completamente jailbroken.

Sistema: kernel R/W, allproc traversal PS5: root + fuera del Jail
06

Fase 5 — ELF loader persistente

loader.js descarga elfldr.elf desde el servidor del PC (vía fetch()), lo copia a memoria, y ejecuta una ROP chain que abre un socket TCP a localhost:9020 y envía el ELF. El loader, al ejecutarse, usa pt_bootstrap() para inyectarse en SceRedisServer vía ptrace y empieza a escuchar en el puerto 9021.

Sistema: fetch + ROP socket send PS5: loader activo en :9021 (persistente)
07

Envío de payloads desde el PC

El usuario ejecuta tools/send_payload.py con la IP de la PS5 y la ruta del payload. El script detecta automáticamente el tipo (ELF/SELF/RAW), valida el archivo, y lo envía al puerto 9021. El ELF loader lo recibe, hace fork(), y ejecuta el payload en un proceso hijo independiente.

Usuario: python3 send_payload.py --file ... PS5: fork() + exec del payload

§07 / guía de ejecución

Pasos del usuario y respuesta del sistema

🧑‍💻 Usuario (PC)

1. Clonar y configurar

Clonar el repositorio y editar exploit/js/loader.js cambiando HOST_IP por la IP del PC en la red local. También editar payload/example/hello.c con la misma IP.

$ git clone https://github.com/TU/ps5-toolkit-11xx $ nano exploit/js/loader.js # HOST_IP = "192.168.1.X"

2. Levantar el servidor HTTP

Ejecutar el servidor en el PC. Mostrará la URL exacta que hay que abrir en la PS5.

$ python3 host/server.py → Abre en PS5: http://192.168.1.X:8000/exploit/index.html

3. Abrir el exploit en la PS5

En la PS5, ir a Ajustes → Usuario → Iniciar Browser (o desde cualquier app que permita abrir el browser). Navegar a la URL del paso anterior y pulsar ▶ Ejecutar exploit.

4. Monitorear los logs (opcional)

En una segunda terminal, escuchar los logs UDP que envía el ELF loader de la PS5 en tiempo real.

$ python3 tools/listen_log.py → Mostrará mensajes del loader cuando esté activo

5. Enviar un payload

Una vez que el exploit complete las 5 fases, usar send_payload.py para enviar cualquier .elf, .bin o .self.

$ python3 tools/send_payload.py \ --host 192.168.1.50 \ --file payloads/mi_payload.elf

6. Compilar el ELF loader (si se modifica)

Si modificas el loader en C, compilarlo con el SDK de PS5 y copiarlo a la carpeta payloads/ del servidor.

$ export PS5_PAYLOAD_SDK=/opt/ps5-payload-sdk $ make -C elfldr/ $ cp elfldr/elfldr.elf payloads/

⚙️ Sistema (PS5)

A. Descarga de los módulos JS

El browser WebKit de la PS5 carga index.html y los 6 scripts JS en orden. Se inicializa la UI con las 5 fases en estado "esperando".

B. Fase 1 — Primitivas de memoria

triggerWebKitBug() dispara el bug de confusión de tipos. La clase Primitives configura el victim ArrayBuffer apuntando su vector pointer a la dirección objetivo en cada lectura/escritura. Se marcan primitivas OK.

C. Fase 2 — Leak + ROP Worker

Se filtra libkBase. Se crea el Worker y se calienta. findWorkerStack() itera la lista de pthreads de libkernel para encontrar el stack del worker. Se sobreescribe el return address y se hace worker.postMessage() para disparar el ROP.

D. Fase 3+4 — Kernel + Jailbreak

El ROP chain ejecuta syscalls: umtx race para kernel UAF, pipe trick para kread8/kwrite8, allproc traversal para el proc actual, parcheo de ucred y cr_prison, desactivación de SCEP en todos los CPUs. La PS5 está root + fuera del Jail.

E. Fase 5 — ELF loader activo

elfldr.elf se descarga del PC, se mapea en memoria RWX vía ROP, y se envía al puerto 9020 (bootstrap). El loader ejecuta pt_bootstrap("SceRedisServer"), inyecta su shellcode en ese proceso, y el puerto 9021 queda activo.

F. Ejecución del payload

Al recibir bytes en :9021, el loader detecta el tipo (magic bytes), hace fork(), y el proceso hijo ejecuta el payload. Si el hijo crashea, el proceso padre del loader sigue activo. Los logs de cada evento se envían por UDP broadcast.

§08 / requisitos

Lo que necesitas antes de empezar

En el PC

Python 3.8+ (servidor + tools) ps5-payload-sdk (compilar el loader C) Ghidra (para encontrar offsets) ROPgadget (buscar gadgets en sprx) Misma red LAN/WiFi que la PS5

En la PS5

Firmware 11.00 (exactamente) Conexión de red (misma LAN que el PC) Acceso al browser (built-in WebKit) NO actualizar (perderías el exploit)
⚠️
Trabajo pendiente crítico:

Las funciones triggerWebKitBug() y leakLibKernelBase() en index.html son las únicas piezas que requieren análisis del binario WebKit de FW 11.00. El resto del scaffold está completamente implementado y las espera. Ver docs/offsets_guide.md para el proceso con Ghidra.

🔎
Offsets en offsets_1100.js:

Los offsets incluidos son aproximaciones basadas en builds anteriores. Deben verificarse contra el dump real del FW 11.00 con Ghidra antes de ejecutar el exploit completo.

§09 / estado del proyecto

¿Qué está listo y qué falta?

  • ✓
    Estructura del proyecto y scaffold completo — Todos los archivos, módulos y su relación están definidos y documentados.
  • ✓
    Clase Int64 + sistema de primitivas — Implementado y funcional. Solo necesita las funciones base del bug para conectar.
  • ✓
    ROPChain builder completo — API fluida, soporte de gadgets, syscalls, llamadas con 6 argumentos, stack pivot en Worker.
  • ✓
    KernelExploit completo — kbase leak, kernel R/W, container escape, root, SCEP disable.
  • ✓
    ELF Loader en C — Parser ELF64, extracción SELF, soporte RAW, fork() por payload, log UDP.
  • ✓
    Bootstrap ptrace en SceRedisServer — Inyección de shellcode x86-64, redirect de RIP, PT_DETACH.
  • ✓
    Herramientas del PC — server.py, send_payload.py, listen_log.py. Todas funcionales como scripts independientes.
  • !
    Offsets para FW 11.00 — Requieren verificación contra dump real del firmware con Ghidra. El archivo offsets_1100.js tiene los placeholders correctamente nombrados y documentados.
  • ✗
    triggerWebKitBug() — La función que dispara el bug específico del FW 11.00. Requiere análisis del binario WebKit con el OSS mirror de Sony + Ghidra. Es el único eslabón faltante para tener el exploit end-to-end.
  • ✗
    leakLibKernelBase() — El leak de la base de libkernel desde el proceso WebKit. Depende de conocer un offset en la GOT de WebKit que apunte a libkernel, lo que requiere análisis del binario.
